# 実験デモ・レポートリポジトリの構成規約（汎用知見）

本ドキュメントは、**テーブルデータ合成（tabular SDG）アルゴリズムの実証評価**で確立した
「実験デモ + 実証評価レポート + 公開サイト」型リポジトリの構成・運用知見を、
任意の合成アルゴリズム評価リポジトリへ転用できる形で整理したものである。
`.claude/skills/` の各スキルはこの規約を前提に整理・レビュー・対応を行う。

> **本テンプレートの位置づけ**: これは `tabular-sdg-skillset` というコピー用テンプレートの一部であり、
> 差分プライバシー（DP）合成の評価（指標 TVD/TSTR/MIA・予算 ε・ベースライン INDEPENDENT 等）を
> 具体例として保持している。CTGAN・TVAE・Copula など DP 以外の手法に適用する場合も、
> 章構成・記述の分離・計画/実行/公開のプロセスはそのまま使える。
> **差し替えが必要なプレースホルダ**（`<owner>/<repo>` 等）と適用手順は、
> テンプレート直下の `README.md` を参照すること。

---

## 1. ディレクトリ構成の原則

```
<repo>/
├── README.md            … 入口。スコープ宣言 + クイックスタート + 構成図 + 公開URL
├── content/             … 公開ページの Markdown ソース（唯一の編集対象）
│   ├── REPORT.md        … レポート本体（論文調・章構成）
│   ├── EXPERIMENTS.md   … 追加実験（本体の発見の深掘り）
│   ├── method-*.md      … 対象の構成要素ごとの解説 + 選定ガイド
│   ├── setup.md / usage.md / reproduce.md / engineering-notes.md
│   │                    … エンジニアリング情報（本体から分離）
├── htmls/               … ビルド成果物（直接編集しない。Pages 公開元）
├── scripts/             … 番号プレフィックスで実行順を表すスクリプト群
│   ├── setup_env.sh / run_all.sh      … 一括セットアップ・一括再現
│   └── 00_..〜NN_..py                 … データ取得→生成→評価→追加実験→ビルド
├── requirements.txt     … 全推移的依存をバージョン固定（再現性の要）
├── patches/             … 上流ライブラリへの最小修正（理由をファイル名と notes に記録）
├── docs/plans/          … 実験計画書（実験ごとに 1 ファイル。実施後は記録を追記して閉じる）
├── data/ outputs/ figures/ experiments/ … 入出力（生成物は再実行で再生成可能に）
└── src/                 … 上流のクローン（git 管理外、setup が取得・パッチ）
```

要点:
- **ソースと成果物を分離する**: `content/`（手で書く）と `htmls/`（ビルドで生成）を混ぜない。
  成果物を直接編集した変更は次のビルドで消える。
- **スクリプトは番号で依存順を表す**: `00→01→02→10→11→03` のように、
  本系列（00〜0N）と追加実験（10〜）とビルド（03）を番号帯で分ける。
- **一括と個別の両方を用意する**: `run_all.sh` で clone-and-run、個別スクリプトでデバッグ。

## 2. 記述の分離原則（最重要）

レポートの信頼性は「どこまでが事実で、どこからが推定か」の区別で決まる。

| 区分 | 内容 | 置き場所 | マーカー |
|---|---|---|---|
| **前提** | 対象のドキュメント・コードから読み取れる仕様 | REPORT.md 第3章相当 | 出典タグ [n] を必ず付す |
| **結果** | 本実験での実測値 | REPORT.md 結果章 / EXPERIMENTS.md | 実行環境・シード・バージョンを明記 |
| **考察** | 実測からの分析・推定 | 考察章 / 各「所見」 | `> 🔎 **考察**` 等のブロックで明示 |
| **課題** | 残課題・適用範囲の限界 | 残課題章 + GitHub Issues | 残課題は Issue 化する |
| **エンジニアリング記録** | 動かすまでの問題と対応 | engineering-notes.md | 「研究知見ではなく一次記録」と位置づけを宣言 |

- 研究的内容（前提・結果・考察）とエンジニアリング情報（環境構築・API 利用・再現手順・実装問題）は
  **ページを分ける**。本体レポートにはリンクだけ残す。
- **トップページ（レポート本体・index）は検証対象を主軸にする**。レポート本体は
  「評価対象ライブラリ（検証対象）で何が分かったか」を主語にして書き、構成要素
  （機構・アルゴリズム・共有コンポーネント等）の**アルゴリズム固有の深掘りは個別ページ
  （`method-*.md` 等）に分離**する。本体には**要約＋個別ページへのリンク**だけを残す。
  - **本体に残す**: 検証対象に固有の発見・前提（例: ある機能が下流有用性に効くという検証結果の要約、
    検証対象のパイプライン由来の挙動、実装成熟度・スコープ・予算と有用性のトレードオフの確認）。
  - **個別ページへ移す**: 構成要素のアルゴリズム由来の性質（例: ある手法が別の手法を上回る理由＝
    その手法固有の設計、ある手法の構造的限界、コンポーネントの計算コスト・パラメータ依存の詳細）。
  - 移設時は、本体側に「要約 1〜2 行＋該当ページへのリンク」を残し、個別ページの既存の実測節へ
    重複・矛盾なく統合する。出典タグ [n]・アンカー・図参照は壊さない。
- 単一シードの数値を断定的に書かない。揺れの定量化（マルチシード mean±std）を別実験として持ち、
  本文では「傾向を読む」よう注記する。
- スコープ外（評価していない機能）は README 冒頭とレポート内の両方で宣言する。

## 3. 公開（ビルド & Pages）の原則

- `scripts/03_build_html.py` 相当のビルダーが `content/*.md` → `htmls/*.html` を生成する。
  ページ一覧・出力名・サブタイトル・ナビは**ビルダー内の PAGES テーブルが単一の真実**。
  ページを追加・改名したら必ず PAGES とナビを更新する。
- 図は base64 埋め込みで自己完結 HTML にし、HTML 単体で共有可能にする。
- 公開は GitHub Actions（`.github/workflows/deploy-pages.yml`）。
  DP 生成などの重い処理は公開ビルドに含めず、Markdown 変換のみで数十秒に収める。
- Markdown 内の相互リンクは公開後の HTML 名（`setup.html` 等）で書く
  （GitHub 上の .md 閲覧では切れるが、公開サイトを正とする）。

## 4. 再現性の原則

- 依存は**推移的依存まで**ロックする（`uv pip compile` 等）。git 依存はコミットハッシュ固定。
- 上流への修正は `patches/*.patch` として最小化し、setup スクリプトが自動適用する。
  パッチの理由・予算等への影響（例: DP 予算消費は不変）を engineering-notes に記録する。
- 外部データの取得元は URL とともに消失リスクを認識し、チェックサム検証・ベンダリングを
  バックログに積む。
- 実行環境（OS・Python・ハードウェア制約）を README と REPORT 冒頭の両方に明記する。

## 5. 課題管理の原則

- 残課題は REPORT の残課題章（読者向けの限界の説明）と
  [GitHub Issues](https://github.com/<owner>/<repo>/issues)（開発者向けの追跡）の二層で持つ。
- 新しい残課題は Issue として起票し、`タイトル / ラベル / 本文（背景・やること・完了基準）` を備える。
  対応済みは Issue を close し、レポート側の残課題章からは該当行を消すか「→ #NN で対応」と追記する。

## 6. 実験の計画・実行の原則

レポート作成と実験は次のライフサイクルで進める。**レポート骨子の検討が先**で、
実験は骨子を段階的に埋めるために行う。

```
report-skeleton（骨子: 章立て・主張・必要な実験の抽出）
  → experiment-plan（計画: 指標の出典・公平な比較条件・業務シナリオ → docs/plans/）
  → experiment-run（実行: uv 固定環境・シード規約・結果のレポート反映）
  → doc-cycle（整理・レビュー・公開）
```

この全体は `research-cycle` スキルが統括する（現在地を判定し、該当プロセスから順次実行）。

### モデルティアの規則（プロバイダ非依存）

役割ごとに要求するモデルの**ティア**を定める。具体的なモデル名はこの規則に書かず、
各エージェント定義（`.claude/agents/*.md`）の `model:` 1 行に集約する
（モデルやプロバイダが変わってもそこだけ差し替えればよい）。

| 役割 | ティア | エージェント | 理由 |
|---|---|---|---|
| 計画（骨子・実験計画・調査・issue 起案） | **上位** | `planner` | 判断の誤りが下流全体に波及する |
| レビュー（指摘の洗い出し・整合点検） | **中位** | `reviewer` | チェックリスト駆動で判断範囲が限定される |
| 実行・実装（スクリプト・環境・実験・ビルド） | **下位** | `implementer` | 計画とテストで制約され、量が多い |

下位ティアの裏付けとして、実行・実装側には機械的な制約（計画書・issue の完了条件・TDD・
計画外の条件変更の禁止）を課す。制約で品質を担保できない作業はティアを上げるのではなく、
まず計画・完了条件の側を具体化する。

### 作業単位の規則（worktree + issue）

- **並行作業は git worktree で行う**。worktree ごとに作業ブランチを切り、
  名前に issue 番号を含める（例: `git worktree add ../<repo>-issue-12 -b issue-12-<slug>`）。
- **worktree には対応する issue が必ず存在する**。issue のない worktree を作らない。
- **issue の内容は計画時に定める**（`experiment-plan` / planner の担当）:
  目的・完了条件（テスト可能な形）・担当エージェント・参照する計画書（`docs/plans/`）を本文に書く。
  既存の GitHub Issue があれば相互参照する。
- **worktree に対応する issue はサブエージェント（implementer）が担当する**。
  issue の完了条件の範囲だけを行い、範囲外は差し戻す。
  コミットメッセージで issue 番号を参照し、完了したら worktree を畳む（`git worktree remove`）。

### 実装の規則（TDD）

- 実装を伴う issue は **TDD** で進める: 完了条件を `tests/`（pytest）の失敗するテストとして
  先に書き、テストが通るように実装し、通った状態でリファクタリングする。
- テストにできない作業（実験実行・ビルド等）は、機械的な検証方法
  （生成物の存在・exit code・件数チェック）を issue の完了条件に定めて代える。

エージェント間の受け渡しは成果物（`docs/plans/` の計画書・issue・指摘リスト）を介して行い、
口頭の文脈に依存させない。

- **骨子先行**: アブストラクトと結論は最後に書く。骨子の TODO 残数で進捗を測る。
- **計画書を残す**: 実験ごとに `docs/plans/<ID>-<slug>.md`（目的・仮説・指標・条件・完了基準）。
  実施後は実行記録を追記して閉じる。計画なしの場当たり実行をしない。
- **指標は既存研究準拠**: 各指標に出典を付け、独自指標には理由を明記する
  （本リポジトリでは TVD・TSTR・MIA など DP 合成データ評価の標準指標に準拠）。
- **公平な比較**: 同一データ・同一前処理・同一プライバシー予算・同一シード集合。
  ベースライン（最単純手法）を必ず含め、チューニングの扱いを全対象で揃える。
- **業務課題を反映した評価**: 想定ユースケース（データ共有・ML 学習用提供等）を 1 行で書き、
  そのユースケースで実際に問われる指標・失敗モードを測る実験を追加する。
- **再現性は uv で担保**: 環境構築は setup スクリプト経由のみ。依存は推移的依存まで固定し、
  実行環境のメタ情報（バージョン・シード）を生成物と一緒に保存する。

## 7. レビュー・対応の進め方

`.claude/skills/doc-cycle` が以下を順次実行する（個別に呼んでもよい）:

1. `report-review` — 前提・結果・考察・課題の分離と整合のレビュー
2. `repro-engineering-review` — 再現手順・依存固定・エンジニアリングノートのレビュー
3. `related-info-review` — 参考文献・上流情報・バックログの整理
4. `publish-check` — content↔htmls↔ナビ↔README の整合確認とビルド検証

各スキルは「指摘リスト（重要度付き）→ ユーザー確認不要な軽微修正は適用 → 残りは報告」の流れで動く。
