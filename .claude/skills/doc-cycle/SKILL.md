# ドキュメント整理・レビューサイクル（オーケストレーター)

ライフサイクル全体の中での位置づけ:
`report-skeleton`（骨子）→ `experiment-plan`（計画）→ `experiment-run`（実行）→ **doc-cycle（整理・レビュー・公開）**。
本スキルは最後の「整理」を担う。骨子や計画の段階で呼ばれた場合は、先行スキルへ誘導する。

`.claude/docs/documentation-conventions.md` の規約に基づき、以下のスキルを**この順で**実行する。
順序に意味がある: 内容（レポート）→ 実体との整合（再現・エンジニアリング）→ 外部情報 → 公開、
の順に確定させると手戻りが少ない。

## 実行手順

1. **準備**: `git status` と直近のコミットを確認し、今回のレビュー範囲
   （全体か、直近の変更に関連するページのみか）を決める。
   引数で範囲指定があればそれに従う（例: `doc-cycle experiments` → EXPERIMENTS 関連のみ）。
2. **report-review** を実行 — 前提・結果・考察・課題の分離と数値整合。
3. **repro-engineering-review** を実行 — 再現手順・依存・パッチ・エンジニアリングノート。
4. **related-info-review** を実行 — 参考文献・出典タグ・上流情報・バックログ。
5. **publish-check** を実行 — content↔ビルダー↔htmls↔README の整合とビルド検証。
6. **横断確認**: 各ステップの修正が他のページとの整合を崩していないか
   （例: report-review で直した数値が method-*.md にも出ていないか）を最後に確認する。
7. **対応**: 軽微な修正はコミットにまとめ、要判断の指摘と新規バックログ候補を統合レポートとして報告する。

各スキルは Skill ツールで呼び出すか、サブエージェントに SKILL.md の手順を渡して並列性より
**順次実行**を優先する（前段の修正が後段の点検対象になるため）。
サブエージェントに委譲する場合は、指摘の洗い出し・軽微修正は `reviewer`（中位ティア）、
ビルド実行や確定済み修正の適用は `implementer`（下位ティア）に分ける
（ティア規則は `.claude/docs/documentation-conventions.md`、定義は `.claude/agents/`）。

## 統合レポートの形式

```
# doc-cycle 統合レポート
## サマリ（1〜3 行: 全体の健全度と最重要の指摘）

## 各レビューの結果
（各スキルの出力をそのまま掲載）

## 横断的な指摘
- 複数ページにまたがる不整合・構造的な改善提案

## 実施した対応
- コミットした修正の一覧

## 要判断事項（ユーザーへの確認が必要なもの）
- [重要度] 指摘・選択肢・推奨
```

## 運用メモ

- 修正コミットは「機械的修正（リンク・数値転記・表記）」と「内容に関わる修正」を分ける。
- 公開（main へのマージ）前には必ず publish-check までを通す。
- サイクルで繰り返し見つかる種類の問題は、規約
  （`.claude/docs/documentation-conventions.md`）か各 SKILL.md のチェックリストに追記して再発を防ぐ。
