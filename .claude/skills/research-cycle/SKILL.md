# 実験レポート作成サイクル（トップレベル・オーケストレーター）

ライフサイクル全体を統括する。各プロセスの実体は個別スキルにあり、本スキルは
**現在地の判定 → 該当スキルの順次呼び出し → 進捗の報告**だけを行う（チェックリストの重複は持たない）。

```
① report-skeleton   骨子: 章立て・主張・必要な実験の抽出
② experiment-plan   計画: 指標の出典・公平な比較条件・業務シナリオ → docs/plans/
③ experiment-run    実行: uv 固定環境・結果のレポート反映（実験ごとに ②③ を反復）
④ doc-cycle         整理: report-review → repro-engineering-review → related-info-review → publish-check
```

## サブエージェント委譲（モデルティアの規則）

役割ごとにティアの異なるエージェントへ委譲する（規則の本体と理由は
`.claude/docs/documentation-conventions.md`「モデルティアの規則」、定義は `.claude/agents/`）:

| プロセス | サブエージェント | ティア |
|---|---|---|
| ① 骨子・② 計画（issue 起案を含む）・調査 | `planner` | 上位 |
| ④ レビュー（指摘の洗い出し・軽微修正） | `reviewer` | 中位 |
| ③ 実行・スクリプト実装（TDD）・ビルド・確定済み修正の適用 | `implementer` | 下位 |

- ②→③ の受け渡しは `docs/plans/` の計画書と **issue** を介する。② で planner が起案した
  issue を作成してから、③ を issue 単位で implementer に委譲する。
- ③ の並行作業は **worktree 単位**で行う（Agent ツールの `isolation: "worktree"` を使うか、
  issue 番号入りの worktree を作って渡す）。1 worktree = 1 issue = 1 サブエージェント。
- ④ では指摘の洗い出しを reviewer に、確定した修正の適用を implementer に分けて委譲する
  （軽微な修正は reviewer / オーケストレーター自身が直してもよい）。

## 手順

1. **現在地の判定**: 以下を確認し、どのプロセスから始めるかを決める。
   - `content/REPORT.md` が無い／TODO だらけ → ① から
   - 骨子はあるが `docs/plans/` に未計画の実験が残っている → ② から
   - 未実施（実行記録のない）計画書がある → ③ から
   - 実験は揃っていて整理・公開が残っている → ④ から
   判定根拠（残 TODO 数・未実施計画の一覧）を最初に報告する。
2. **順次実行**: 現在地から ①→②→③→④ の順に該当スキルを呼び出す。
   ②③は実験単位で反復し、1 実験を計画→実行→レポート反映まで通してから次の実験に進む
   （全計画を先に作ってから一括実行すると、前の実験の発見が次の計画に反映できない）。
3. **中断点**: 次の場合は止めてユーザーに確認する。
   - 骨子の主張・スコープの確定（①の出口）
   - 計画の公平性・指標選定に判断が割れる論点（②の出口）
   - 長時間（目安 30 分超）の実行や計算資源を大きく使う実行の開始前
4. **完了**: ④ の doc-cycle 統合レポートに、サイクル全体の進捗
   （完了した実験 / 残りの計画 / 骨子の残 TODO）を併記して報告する。

## 進捗報告の形式

```
# research-cycle 進捗
- 現在地: stage N（report-skeleton 基準）/ プロセス ①〜④
- 完了した実験: <ID> ...（各 1 行サマリ）
- 未実施の計画: docs/plans/...
- 骨子の残 TODO: n 件
- 次のアクション: ...
```
