# 実験計画（公平な比較と業務シナリオの設計）

本プロセスは判断が中心のため、サブエージェントに委譲する場合は
`planner`（上位ティア、`.claude/agents/planner.md`）を使う。

`report-skeleton` が出力した「必要な実験一覧」を入力に、実行可能な計画書を作る。
計画は `docs/plans/<ID>-<slug>.md` に置き、実行（`experiment-run`）と結果整理（`report-review`）から参照する。

## 計画書テンプレート

```markdown
# 実験<ID>: <タイトル>
- 目的: レポートのどの主張を支えるか（章・節への参照）
- 仮説: 期待する結果と、棄却された場合の解釈
- 指標: 名称・定義・**出典（既存研究・先行評価のどれに準拠するか）**
- 比較条件: 対象（機構・手法）× パラメータ（ε 等）× データ。全対象に共通の条件を明記
- 公平性の担保: 同一データ・同一前処理・同一予算・同一シード集合・チューニングの扱い
- 実行: スクリプト番号・所要時間目安・シード数
- 完了基準: 何が得られたら結果章に書けるか
```

## チェックリスト

### 指標の選定（既存研究準拠）
- [ ] 各指標に出典があるか（例: 1-way/2-way TVD・TSTR・MIA は DP 合成データ評価の標準指標）
- [ ] 独自指標・変形を使う場合、その理由と既存指標との関係が書かれているか
- [ ] 指標が主張と対応しているか（分布再現性 / 下流タスク有用性 / 経験的プライバシー のどれを測るか）

### 比較の公平性
- [ ] 全対象に同一のデータ・前処理・サンプリング・プライバシー予算を適用しているか
- [ ] 一方にだけ有利なチューニング（ハイパーパラメータ調整）をしていないか。する場合は全対象に同じ手続きか
- [ ] 乱数シードは複数用意し、対象間で同じシード集合を使うか（計算コストで数を変える場合は明記)
- [ ] ベースライン（最も単純な手法。本リポジトリでは INDEPENDENT）が含まれているか
- [ ] 「対象の既定値 vs 推奨値」のどちらで比較するかを宣言しているか

### 業務課題を反映した評価シナリオ
- [ ] 想定ユースケース（例: 機微データの部署間共有・外部提供・ML 学習用データ提供）を 1 行で書いたか
- [ ] そのユースケースで実際に問われる評価になっているか
      （例: 下流分類タスク=TSTR、攻撃耐性=MIA、集計レポートの一致=マージナル誤差）
- [ ] 失敗モード（業務で困る壊れ方。例: 特定列の分布崩壊・希少カテゴリの消失）を測る実験があるか
- [ ] 結果を業務判断に翻訳する一文が用意できるか（「この条件なら共有可否の判断材料になる」等）

### 実行可能性
- [ ] 所要時間・計算資源の見積もりがあるか（重い対象はシード数を減らす等の調整を計画段階で決める）
- [ ] 既存スクリプトの拡張で済むか、新スクリプト（次の番号帯）が要るか
- [ ] 中間生成物の保存先（outputs/ experiments/ figures/）が決まっているか

### 作業 issue の起案（実行への受け渡し）

計画が固まったら、worktree で実施する**作業単位ごとに issue を起案**する
（規則は規約「作業単位の規則（worktree + issue）」）。issue 本文に含めるもの:

- [ ] 目的（計画書 `docs/plans/<ID>-...` への参照）
- [ ] 完了条件（テスト可能な形。実装を伴うなら TDD のテスト観点、実行のみなら生成物・exit code 等の機械的検証）
- [ ] 担当エージェント（通常 `implementer`）と想定 worktree／ブランチ名（issue 番号入り）
- [ ] 既存 GitHub Issue との対応（既存課題の具体化なら相互参照）

## 出力形式

```
## experiment-plan 結果
### 作成/更新した計画書
- docs/plans/<ID>-<slug>.md
### 公平性・指標に関する判断（要確認事項）
- [重要度] 論点と推奨
```
