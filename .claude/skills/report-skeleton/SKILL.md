# レポート骨子の設計（実験より先に書く）

進め方の原則: **レポート骨子の検討が先。実験はそれを埋めるために行う。**
先に「何を主張したいか・そのために何を測る必要があるか」を骨子として固めることで、
不要な実験を避け、公平な比較条件を実験前に確定できる。

## 手順

1. 評価対象（ライブラリ・手法）の一次情報（README・ドキュメント・コード）をざっと読み、
   スコープ（評価する範囲・しない範囲）を 2〜3 行で言語化する。
2. 標準の章構成で骨子を作る:
   アブストラクト → イントロ（貢献 3 点）→ 関連技術・前提 → 実験設定 → 結果 → 考察 → 残課題 → 結論 → 再現補足 → 参考文献。
3. 各章に「その章で主張したいこと 1〜2 行」と、根拠として必要な実験・指標を書き込む。
   未確定の箇所は `<!-- TODO(stage-N): ... -->` コメントで埋める。
4. 骨子から**必要な実験の一覧**（実験 ID・目的・指標・条件）を抽出し、
   `experiment-plan` スキルへの入力とする。
5. 骨子を `content/REPORT.md` として配置（既存レポートの再構成なら差分提案に留める）。

## 段階的に埋める順序（stage）

| stage | 埋める章 | 完了条件 |
|---|---|---|
| 1 骨子 | 全章の見出し + 各章の主張 1 行 + TODO | 必要な実験一覧が抽出できる |
| 2 前提 | 関連技術・前提（出典タグ付き）・実験設定 | 比較条件・指標が出典付きで確定 |
| 3 結果 | 結果章（実験が終わったものから順に） | 図表 + 実行条件の明記 |
| 4 考察 | 考察・残課題（根拠となる結果を明示） | report-review を通過 |
| 5 仕上げ | アブストラクト・結論（最後に書く） | doc-cycle を通過 |

- アブストラクトと結論は**最後**に書く（結果が出る前に書くと過剰な約束になる）。
- stage が進むたびに該当 TODO を消し込み、残 TODO 数で進捗が分かる状態を保つ。

## 骨子レビューのチェックリスト

- [ ] 貢献（イントロの箇条書き）と章構成が 1 対 1 で対応しているか
- [ ] 各主張に対して、それを支持する実験・指標が割り当てられているか
- [ ] 指標は既存研究に準拠しているか（独自指標を使う場合は理由を明記）
- [ ] スコープ外の宣言があるか（README と一致させる）
- [ ] 業務課題を反映した評価シナリオ（下流タスク・プライバシー攻撃等）が結果章に席を持っているか
- [ ] トップページ（レポート本体）が検証対象を主軸に書けているか（構成要素のアルゴリズム固有の深掘りは個別ページに分離し、本体は要約＋リンクに留める）

## 出力形式

骨子ファイル本体に加えて、抽出した実験一覧を報告する:

```
## report-skeleton 結果
### 骨子: content/REPORT.md（TODO n 件）
### 必要な実験一覧（experiment-plan への入力）
| ID | 目的（支える主張） | 指標（出典） | 条件 |
```
